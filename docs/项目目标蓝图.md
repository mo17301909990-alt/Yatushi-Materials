## 二、四大战略目标

### 🎯 目标 1：AI 对话即匹配（智能匹配升级）

**现状**：用户通过左侧表单选条件（产品类型→图案→烫金类型→纸性能），系统查兼容性规则表 + 排序出结果。AI 目前充当"翻译层"——把自然语言转成结构化参数，再走同一套查询。

**目标态**：

```
用户说："我要烫精装书封面，哑金，耐磨点"
    ↓
AI 理解意图 + 上下文推理
    ├─ 识别：产品类型=精平装/咭书，烫金类型=平面烫金
    ├─ 推理：哑金需用普通耐磨纸
    ├─ 查规则表：该组合是否兼容（V/▷/X）
    └─ 排序：按价格匹配度 + 热度 + 用户偏好
    ↓
输出：推荐物料列表 + 注意事项卡片
```

**关键里程碑**：

| 阶段 | 内容 | 优先级 |
|------|------|--------|
| P1.1 ✅ | 关键词兼容性搜索（`/compatibility/query`） | 已完成 |
| P1.2 | 自然语言→参数解析准确率提升（从 prompt 硬调到 Fine-tuning） | 高 |
| P1.3 | 多轮对话上下文累积（Pinia store 已搭骨架，需深化） | 高 |
| P1.4 | 模糊表述推理（"哑金"="普通耐磨哑光金纸"级的行业知识映射） | 中 |
| P1.5 | 跨工序联合推荐（烫金 → 过胶 → 丝印 一站式出方案） | 中 |
| P1.6 | Copilot 主动反问补全条件（用户说一半，AI 追问缺什么） | 低 |

### 🎯 目标 2：Agent 自主修改后台参数

**现状**：所有业务数据变更（兼容性规则、价格、物料配置、权限）都靠管理员手动点页面 CRUD。

**目标态**：

```
用户说："把 XX 型号的耐磨纸换成高耐磨"
    ↓
Agent 接收到指令
    ├─ 1. 确认权限（操作用户是否有修改兼容性规则的权限）
    ├─ 2. 识别变更内容（哪个表、哪个字段、新旧值）
    ├─ 3. 检查影响范围（这条规则变更会影响多少产品匹配）
    ├─ 4. 生成变更预览（Diff/影响分析）
    ├─ 5. 提请确认（"以下 3 条规则会变更，是否执行？"）
    └─ 6. 执行变更 + 记录审计日志 + 通知相关人员
```

**关键能力建设**：

| 能力 | 说明 | 优先级 |
|------|------|--------|
| 写操作 API 层 | 所有 CRUD 操作需有 Agent 可调用的统一接口 | 高 |
| 变更预览与 Diff | 执行前展示影响范围（类 git diff） | 高 |
| 权限管控 | Agent 继承用户权限，越权操作拒绝 | 高 |
| 操作审计 | 自动写 `admin_operation_log`，含 Agent 执行轨迹 | 高 |
| 审批流 | 高风险变更需二级审批（类飞书审批） | 中 |
| 回滚能力 | 变更失败自动回滚 / 手动一键回滚 | 中 |
| 自然语言→SQL/API | Agent 理解业务语义，自动映射到正确的表和字段 | 中 |
| 变更通知 | 变更完成自动通知相关用户（公告/消息中心） | 低 |

### 🎯 目标 4：全工序资料数字化（ETL 数据 Pipeline）

**现状**：烫金（烫金纸）数据已完备，但硅胶、UV油墨、后加工、LEO纸品、多份烫后加工指引书都还躺在 G 盘的 Excel/Word 里。

**目标态**：所有工序的原始资料都有一套自动化的数据管道：

```
G:\雅图仕\download\{品类}\*.xlsx / *.docx
    ↓ 格式解析器（xlsx/docx 自适应）
HeaderDetector 自动定位表头
    ↓ Schema Handler（策略模式）
产品表 + 兼容性表
    ↓ 验证与质量检查
人工抽检确认 ✅
    ↓
支持线上匹配
```

**数据管线清单**：

| 品类 | 源文件 | 格式 | 处理器状态 | 优先级 |
|------|--------|------|-----------|--------|
| **硅胶** 13 类 | Glow Ink, White UV, Glittering Star... | xlsx | ✅ `silicone` handler 就绪，未运行 | 🔴 高 |
| **UV油墨** 2 类 | 哑光UV油、哑光水油 | xlsx | ✅ `uv_oil_matte` / `water_varnish` handler 就绪，未运行 | 🔴 高 |
| **后加工资料** | 后加工资料标准书 | xlsx | ✅ `lamination` handler 就绪，未运行 | 🟡 中 |
| **烫金指引书 ×9** | 常用界面指引书、烫后加工系列 | docx | ❌ 需新增 docx 解析器 | 🟡 中 |
| **LEO纸品 ×4** | LEO品牌纸、书板贴纸、平面/非平面贴纸 | xlsx | ❌ 需新 handler | 🟡 中 |
| **JP750S** | 印刷后加工工艺标准书 | docx | ❌ 需新 handler | 🟢 低 |

**后续数据质量机制**：

- 导入后自动执行 `SELECT COUNT(*)` 校验行数
- 字段完整性检查（Null 比例警戒线）
- 兼容性规则一致性校验（无矛盾判定）
- 人工抽检报告 + 审计日志

## 三、技术架构演进

### 当前架构

```
用户 → 浏览器(Vue3)
         ↓
    Spring Boot REST API
    ├─ 传统 CRUD 控制器
    ├─ AI Copilot (DashScope)
    │    └─ AiChatOrchestrator（3层上下文注入）
    └─ 匹配引擎
         └─ GoldFoilProductService（规则表 + 匹配度计算）
         ↓
    PostgreSQL（烫金数据完备，其余品类空表）
```

### 目标架构

```
用户 → 浏览器(Vue3) ← 移动端
         ↓
    API Gateway
    ├─ 传统 CRUD 控制器 ← 目标3: 飞书级管理页面
    ├─ Agent Runtime          ← 目标2: 自主修改参数
    │    ├─ 意图识别 (IntentDetector)
    │    ├─ 知识检索 (RAG)
    │    ├─ 工具调用 (Tool Use)
    │    │    ├─ 读：规则查询、产品搜索、统计报表
    │    │    └─ 写：规则变更、配置修改、数据维护
    │    └─ 审批/确认回调
    ├─ AI Copilot             ← 目标1: 对话即匹配
    │    ├─ 多轮对话管理
    │    ├─ 上下文推理
    │    └─ 主动追问补全
    └─ ETL Pipeline           ← 目标4: 数据数字化
         ├─ xlsx 解析器 (openpyxl)
         ├─ docx 解析器 (python-docx)
         ├─ HeaderDetector 自动定位
         └─ Schema Handler 策略
         ↓
    PostgreSQL（全品类数据完备）
```

## 六、约束与注意事项

1. **POI/EasyExcel 版本敏感**：pom.xml 中 POI 锁定 5.2.3 + xmlbeans 5.1.1，依赖冲突会抛 `NoSuchFieldError`
2. **生产数据库不可达**：当前仅能操作本地 PostgreSQL 16 测试库（gold_foil_db）
3. **AI 依赖通义千问 DashScope**：API Key 走环境变量 `DASHSCOPE_API_KEY`
4. **知识库 RAG**：工厂指引书 #N-BF-008 ~ #N-BF-013 已萃取为核心知识嵌入系统
5. **ETL 数据源**：原始资料位于 `G:\雅图仕\download\`，含 xlsx 和 docx 两种格式
6. **Git 工作流**：feature 分支 → PR → main，禁用 force push
7. **所有 AI 写操作必须有人审批**：Agent 不能直接修改业务数据，必须通过变更预览 → 确认的闭环
