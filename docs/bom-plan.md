# 雅图仕智能 BOM 系统 — 提速方案

> 编制角色：BOM 系统部门负责人
> 背景：鹤山雅图仕，亚洲最大单体印刷厂，8000+ 员工，年产值 21.6 亿
> 当前瓶颈：32 大类产品，客户 PO → 生产作业单，1 人/天仅完成 2 个 BOM

---

## 一、现状与目标

### 1.1 业务规模

| 维度 | 数据 |
|------|------|
| 产品大类 | 32 类（精装书/平装书/立体书/儿童书/贺卡/礼盒/纸袋/游戏套装…） |
| 年产值 | 21.6 亿元 |
| 员工 | 8,000+ |
| 客户 | 100% 出口，欧/美/澳/亚 |
| 当前 BOM 速度 | **2 个/人/天** |
| 目标 BOM 速度 | **50+ 个/人/天**（25x 提升） |

### 1.2 全流程分解（从 PO 到作业单）

```
客户 PO 单 (PDF/传真/邮件)
  │
  ▼
① 订单解析 ← 人工读 PO → 提取：品名/数量/规格/纸张/工艺要求
  │  耗时：30min-2h（取决于 PO 复杂度）
  ▼
② 物料选型 ← 确定具体物料：纸/烫金纸/UV油/丝印油墨/过胶膜…
  │  耗时：20min-1h
  ▼
③ 工艺定序 ← 前加工→主工艺→后加工→包装
  │  耗时：30min-2h（复杂产品要反复确认）
  ▼
④ 设备分配 ← 匹配到具体机台
  │  耗时：10min
  ▼
⑤ 用料核算 ← 算用量/损耗/拼版
  │  耗时：20min
  ▼
⑥ 输出作业单 ← 生产部门可执行的文件
  │  耗时：10min
  └── 总计：**2-7 小时 / BOM**
```

### 1.3 当前系统能做什么

已有物料匹配系统（YTS）覆盖了 **② 物料选型** 环节的烫金部分——兼容性规则已经线上运行。丝印/UV 油/过胶的数据已入库，匹配规则可扩展。

**但这个系统目前只做了 32 大类中的 4-5 类的"物料 vs 物料"匹配。** 要解决一天 2 个 BOM 的问题，需要的是 **全品类 × 全流程**。

## 二、瓶颈分析（为什么一天只能做 2 个）

### 2.1 各环节耗时

| 环节 | 占 BOM 总时间 | 可自动化程度 | 优先级 |
|------|-------------|-------------|--------|
| ① PO 解析 | 25% | ⭐⭐（需要 NLP/模板） | P1 |
| ② 物料选型 | 20% | ⭐⭐⭐⭐（规则引擎） | **P0** |
| ③ 工艺定序 | 30% | ⭐⭐⭐（规则+模板） | **P0** |
| ④ 设备分配 | 5% | ⭐⭐⭐⭐ | P2 |
| ⑤ 用料核算 | 10% | ⭐⭐⭐ | P1 |
| ⑥ 输出作业单 | **10% → 实际 40%** | ⭐⭐⭐⭐⭐ | **P0** |

### 2.2 真正的问题：不是"不会算"，是"算出来要填半天"

现场跟单员的真实工作流不是"思考"，而是**填表和复制粘贴**：

```
算出一个物料 → 切到 ERP 系统 → 打开 BOM 维护界面
  → 找到对应字段 → 敲进去 → 切回来算下一个
  → 算出一道工序 → 切到 ERP → 填工序页签 → 切回来
  → 算完用量 → 切到 ERP → 填物料用量 → 切回来
  → 算清各道工序名称 → 逐个填到 ERP → 切回来
  
每次切换 5-10 秒 + 找到对应字段 10-30 秒
一个 BOM 30 个字段 × 40 次切换 = 20+ 分钟浪费在"切界面"上
```

**实际拆解一个精装书 BOM 的时间：**

```
PO 阅读理解:       30min  ← 不可省，但可减少
物料查规则:        30min  ← 系统已有，但结果要手填
工艺定序思考:      30min  ← 靠经验，系统暂无
ERP 数据录入:      **120min**  ← 纯机械劳动，最该砍掉
核对校验:          30min  ← 因为手填怕填错
─────────────────────────
总计:             4h/BOM
```

**结论：** 即使算得再快，只要结果要手填到 ERP，就快不了。**50% 的时间花在"把数据从一个系统搬到另一个系统"。**

### 2.3 核心矛盾

**人做 BOM 慢的原因不是"不懂"，而是"要查的东西太多 + 要填的东西更多"：**

- 32 类产品 × 每类 10+ 种工艺组合 = 300+ 种可能流程
- 每种组合涉及 5-15 个物料，每个物料有兼容性约束
- 资深师傅靠记忆和经验做判断，新人工需要 3-6 个月培训
- **即使算出来，ERP 系统分散在 10+ 个界面，需要逐字段手动录入**
- 一个复杂产品（如立体书+发声器件）的工艺路线可能有 20+ 道工序

**一天 2 个 = 系统的认知极限 + 界面极限。**

## 三、提速方案 — 三层递进

### 第一阶段：物料匹配全品类覆盖（4-6 周）— 3x 提速

**目标：** 从烫金扩展到 32 大类的物料兼容性匹配。物料选型从 1h → 5min。

| 品类 | 当前状态 | 所需工作 |
|------|---------|---------|
| 烫金 | ✅ 已线上 | — |
| 丝印 14 类 | ✅ 数据已入库 | 匹配规则配置 |
| UV 油 | ✅ 数据已入库 | 匹配规则配置 |
| 过胶/裱纸 | ✅ 数据已入库 | 匹配规则配置 |
| LEO 纸品 | ✅ 数据已入库 | 匹配规则配置 |
| **书籍类**（精装/平装/立体/儿童） | ❌ 未入库 | ETL 数据导入 + 规则配置 |
| **包装类**（礼盒/酒盒） | ❌ 未入库 | ETL 数据导入 + 规则配置 |
| **贺卡/纸袋/文具/游戏** | ❌ 未入库 | ETL 数据导入 + 规则配置 |

**工作量：** 已有框架和 ETL 管线，新品类主要是数据导入 + 规则配置。每类约 2-3 天。

### 第二阶段：工序兼容性 + 工艺模板库（4-6 周）— 8x 提速

**目标：** 解决③工艺定序——BOM 最慢的环节。从 2h → 10min。

#### 3.1 工序兼容性规则

现有系统做的是"物料 vs 物料"兼容。BOM 需要的是"工序 vs 工序"兼容：

```
烫金 → UV 油      ?  可不可以先烫金再过 UV油？
烫金 → 过胶       ?  可不可以先烫金再过胶？
丝印 → 烫金       ?  顺序有没有限制？
UV 油 → 裱纸      ?  需要间隔时间吗？
```

**核心发现：** 这套逻辑**跟物料匹配完全一样**——只是匹配对象从"物料"换成"工序"。同一个规则引擎可以覆盖。

#### 3.2 工艺模板库

32 类产品，每类有 3-8 种典型工艺路线。把这些提炼成模板：

```
产品类型: 精装书
典型工艺:
  ├─ 基础: 印刷 → 折页 → 配页 → 锁线 → 上壳 → 压槽 → 包装
  ├─ +过胶: 印刷 → 过胶 → 折页 → 配页 → 锁线 → 上壳 → 压槽 → 包装
  └─ +烫金: 印刷 → 烫金 → 过胶 → 折页 → 配页 → 锁线 → 上壳 → 压槽 → 包装

产品类型: 礼盒
典型工艺:
  ├─ 基础: 印刷 → 裱纸 → 模切 → 糊盒 → 包装
  ├─ +UV: 印刷 → UV → 裱纸 → 模切 → 糊盒 → 包装
  └─ +烫金+击凸: 印刷 → 烫金 → 击凸 → UV → 裱纸 → 模切 → 糊盒 → 包装
```

**模板选好 → 规则引擎自动校验兼容性 → 微调 → BOM 完成。**

#### 3.3 前加工/后加工自动关联

现有数据已经区分了"前加工工序"和"后加工工序"。建立自动关联规则：

```
前加工选项 → 限制后加工选项
  "过普通膜"   → 后加工可选: 烫金/丝印/UV (但温度受限)
  "过UV膜"     → 后加工可选: UV (不可烫金)
  "不加工"     → 后加工可选: 全部
```

### 第三阶段：AI BOM 编排 + PO 解析（6-8 周）— 25x 提速

**目标：** 输入 PO → AI 推荐完整 BOM，人工确认即可。一个 BOM 从 4h → 10min。

#### 3.1 AI BOM 编排

基于现有 AI Copilot（DashScope）扩展：

```mermaid
输入: "5000 张精装贺卡，157g 铜版纸，烫金+击凸，过哑膜"
  │
  ▼
① IntentDetector 识别：
   类型=贺卡, 工艺=烫金+击凸+过膜, 数量=5000
  │
  ▼
② SkillRouter 匹配工艺模板：
   贺卡_烫金击凸过膜模板
  │
  ▼
③ 从数据库加载：
   - 物料匹配结果（金箔纸 A 适配 ✓）
   - 工序兼容性校验（烫金→击凸→过膜 顺序 OK ✓）
   - 设备分配（烫金机 T-03 可用 ✓）
  │
  ▼
④ 输出 BOM 草案 → 人工确认 → 生产作业单
```

**已具备的 AI 基础设施：**
- AiChatOrchestrator（3 层渐进上下文）✅
- IntentDetector + SkillRouter ✅
- DashScope API 连接 ✅
- 已有物料匹配数据库 ✅

**需要新增的：**
- BOM 模板库（32 类 × 5-8 模板/类 = 约 200 模板）
- 工序兼容性规则表
- BOM 草案推荐 Prompt

#### 3.2 PO 结构化入口

最简单的方式：**做一个订单录入界面，不搞 OCR/NLP**

```
客户 PO (PDF/图片/邮件)
  │
  ├─ 高频客户 → 建产品模板（客户 A 的"贺卡 NO.123" = 已知 BOM）
  ├─ 中频客户 → 下拉选择（品类+工艺+纸张+数量）
  └─ 低频客户 → 自由输入 + AI 解析
```

不需要完美解析 PDF——让跟单员花 2 分钟选参数，比花 2 小时读 PO 做 BOM 快 60 倍。

## 四、技术实现路径

### 4.1 根治"复制粘贴"：BOM 一站式录入界面

**核心原则：匹配系统直接输出 BOM，不让人的手碰 ERP。**

```
现状：
  匹配系统(结果) → 人看 → 切 ERP → 找字段 → 敲 → 切回来 → 重复 x 40次

目标：
  匹配系统(结果) → 一键填入 BOM 界面 → 确认 → 推送 ERP
                       ↓
                 全过程不离开一个界面，不复制粘贴一行
```

#### BOM 录入界面的设计

以精装书为例，跟单员面对的不应该是 10 个 ERP 界面页签，而是一个**按"工序流"组织的单页表单**：

```
客户: ABC Publisher              PO#: PO-2026-0628
产品: 精装儿童书 A5/32p/157g铜版/过哑膜/锁线精装

-- 纸张 ----------
内页: [157g 铜版纸]      封面: [250g 铜版纸]
裱纸: [无]               <- 匹配系统自动推荐，下拉确认

-- 印刷 ----------
印刷方式: [胶印]          色数: [4C+0C]

-- 表面处理 -------
过胶: [哑膜]              UV: [无]
烫金: [金箔纸 A]          位置: [封面书名]
         ^ 匹配系统根据兼容性自动过滤可选列表

-- 装订 ----------
装订方式: [锁线精装]      书脊: [圆脊]

-- 后加工 ---------
模切: [无]               击凸: [封面书名]

-- 包装 ----------
包装方式: [热塑]          每箱数量: [50]

------------------------------------------------
工序流预览: 印刷 -> 过哑膜 -> 烫金 -> 锁线精装 -> 包装
兼容性校验: 全部通过
总物料数: 7               预估工时: 3.5h
------------------------------------------------
[保存草稿]    [提交 BOM -> 推送 ERP]
```

**关键设计原则：**
1. **一页到底** — 不翻页签，不切界面。纸张/印刷/表面处理/装订/后加工按生产流纵向排列
2. **下拉选择** — 所有字段都是选择，不是填空。匹配系统自动过滤可选值
3. **实时校验** — 选了烫金，系统自动检查烫金纸 + 纸张 + 过胶的兼容性，不兼容的选项直接灰掉
4. **一键推送** — BOM 确认后，通过 API 写入 ERP 的对应界面（不经过人工复制粘贴）

#### ERP 对接策略

不改造现有 ERP，而是在 BOM 系统里**建好完整数据后，调用 ERP 的 API 写入**：

```
BOM 系统 (新)
  +-- 物料匹配     <- 已有 YTS
  +-- 工序兼容性   <- 新建规则表
  +-- 工艺模板     <- 新建模板库
  +-- BOM 录入界面  <- 新建 Vue 页面
       |
       v (API 批量写入)
ERP 系统 (现有)
  +-- 订单模块
  +-- BOM 维护界面 (不再人工操作)
  +-- 物料模块
  +-- 工艺路线模块
```

**如果 ERP 没有开放 API：** BOM 系统输出标准格式（JSON/Excel），用脚本批量导入或生成 ERP 能识别的中间表。至少省掉"逐个字段敲"的 120 分钟。

### 4.2 已有资产的复用

### 4.4 排期

总计：10 周

## 六、MVP 建议（3 周）

如果资源有限，先做最小验证：

```
选 2 个高频品类：精装书 + 礼盒（占产量 40%+）
  → 3 周做完 → 验证 BOM 速度提升
  → 效果好 → 扩展到 32 类
```

**3 周 MVP 内容：**
1. 精装书 + 礼盒的工序兼容性规则（匹配现有规则引擎）
2. 各 5 个典型工艺模板
3. **BOM 一站式录入界面（核心）** — 一页到底，下拉选择，自动校验
4. **ERP 对接** — JSON 导出（API 写入视 ERP 开放程度而定）
5. 目标：从 2 个/天 → 10 个/天（限精装书+礼盒）
   其中 ERP 录入时间从 120min → 5min
