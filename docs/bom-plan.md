# 雅图仕智能 BOM 系统 — 提速方案

> 编制角色：BOM 系统部门负责人
> 背景：鹤山雅图仕，亚洲最大单体印刷厂，8000+ 员工，年产值 21.6 亿
> 当前瓶颈：32 大类产品，客户 PO → 生产作业单，1 人/天仅完成 2 个 BOM

---

## 一、现状与目标

### 1.1 业务规模

| 维度 | 数据 |
|------|------|
| 产品大类 | 32 类（精装书/平装书/立体书/儿童书/贺卡/礼盒/纸袋/游戏套装…） |
| 年产值 | 21.6 亿元 |
| 员工 | 8,000+ |
| 客户 | 100% 出口，欧/美/澳/亚 |
| 当前 BOM 速度 | **2 个/人/天** |
| 目标 BOM 速度 | **50+ 个/人/天**（25x 提升） |

### 1.2 全流程分解（从 PO 到作业单）

```
客户 PO 单 (PDF/传真/邮件)
  │
  ▼
① 订单解析 ← 人工读 PO → 提取：品名/数量/规格/纸张/工艺要求
  │  耗时：30min-2h（取决于 PO 复杂度）
  ▼
② 物料选型 ← 确定具体物料：纸/烫金纸/UV油/丝印油墨/过胶膜…
  │  耗时：20min-1h
  ▼
③ 工艺定序 ← 前加工→主工艺→后加工→包装
  │  耗时：30min-2h（复杂产品要反复确认）
  ▼
④ 设备分配 ← 匹配到具体机台
  │  耗时：10min
  ▼
⑤ 用料核算 ← 算用量/损耗/拼版
  │  耗时：20min
  ▼
⑥ 输出作业单 ← 生产部门可执行的文件
  │  耗时：10min
  └── 总计：**2-7 小时 / BOM**
```

### 1.3 当前系统能做什么

已有物料匹配系统（YTS）覆盖了 **② 物料选型** 环节的烫金部分——兼容性规则已经线上运行。丝印/UV 油/过胶的数据已入库，匹配规则可扩展。

**但这个系统目前只做了 32 大类中的 4-5 类的"物料 vs 物料"匹配。** 要解决一天 2 个 BOM 的问题，需要的是 **全品类 × 全流程**。

---

## 二、瓶颈分析（为什么一天只能做 2 个）

### 2.1 各环节耗时

| 环节 | 占 BOM 总时间 | 可自动化程度 | 优先级 |
|------|-------------|-------------|--------|
| ① PO 解析 | 25% | ⭐⭐（需要 NLP/模板） | P1 |
| ② 物料选型 | 20% | ⭐⭐⭐⭐（规则引擎） | **P0** |
| ③ 工艺定序 | 30% | ⭐⭐⭐（规则+模板） | **P0** |
| ④ 设备分配 | 5% | ⭐⭐⭐⭐ | P2 |
| ⑤ 用料核算 | 10% | ⭐⭐⭐ | P1 |
| ⑥ 输出作业单 | 5% | ⭐⭐⭐⭐ | P1 |

### 2.2 核心矛盾

**人做 BOM 慢的原因不是"不懂"，而是"要查的东西太多"：**

- 32 类产品 × 每类 10+ 种工艺组合 = 300+ 种可能流程
- 每种组合涉及 5-15 个物料，每个物料有兼容性约束
- 资深师傅靠记忆和经验做判断，新人工需要 3-6 个月培训
- 一个复杂产品（如立体书+发声器件）的工艺路线可能有 20+ 道工序

**一天 2 个 = 人的认知极限。这问题不是加人能解决的——需要系统替代记忆。**

## 三、提速方案 — 三层递进

### 第一阶段：物料匹配全品类覆盖（4-6 周）— 3x 提速

**目标：** 从烫金扩展到 32 大类的物料兼容性匹配。物料选型从 1h → 5min。

| 品类 | 当前状态 | 所需工作 |
|------|---------|---------|
| 烫金 | ✅ 已线上 | — |
| 丝印 14 类 | ✅ 数据已入库 | 匹配规则配置 |
| UV 油 | ✅ 数据已入库 | 匹配规则配置 |
| 过胶/裱纸 | ✅ 数据已入库 | 匹配规则配置 |
| LEO 纸品 | ✅ 数据已入库 | 匹配规则配置 |
| **书籍类**（精装/平装/立体/儿童） | ❌ 未入库 | ETL 数据导入 + 规则配置 |
| **包装类**（礼盒/酒盒） | ❌ 未入库 | ETL 数据导入 + 规则配置 |
| **贺卡/纸袋/文具/游戏** | ❌ 未入库 | ETL 数据导入 + 规则配置 |

**工作量：** 已有框架和 ETL 管线，新品类主要是数据导入 + 规则配置。每类约 2-3 天。

### 第二阶段：工序兼容性 + 工艺模板库（4-6 周）— 8x 提速

**目标：** 解决③工艺定序——BOM 最慢的环节。从 2h → 10min。

#### 3.1 工序兼容性规则

现有系统做的是"物料 vs 物料"兼容。BOM 需要的是"工序 vs 工序"兼容：

```
烫金 → UV 油      ?  可不可以先烫金再过 UV油？
烫金 → 过胶       ?  可不可以先烫金再过胶？
丝印 → 烫金       ?  顺序有没有限制？
UV 油 → 裱纸      ?  需要间隔时间吗？
```

**核心发现：** 这套逻辑**跟物料匹配完全一样**——只是匹配对象从"物料"换成"工序"。同一个规则引擎可以覆盖。

#### 3.2 工艺模板库

32 类产品，每类有 3-8 种典型工艺路线。把这些提炼成模板：

```
产品类型: 精装书
典型工艺:
  ├─ 基础: 印刷 → 折页 → 配页 → 锁线 → 上壳 → 压槽 → 包装
  ├─ +过胶: 印刷 → 过胶 → 折页 → 配页 → 锁线 → 上壳 → 压槽 → 包装
  └─ +烫金: 印刷 → 烫金 → 过胶 → 折页 → 配页 → 锁线 → 上壳 → 压槽 → 包装

产品类型: 礼盒
典型工艺:
  ├─ 基础: 印刷 → 裱纸 → 模切 → 糊盒 → 包装
  ├─ +UV: 印刷 → UV → 裱纸 → 模切 → 糊盒 → 包装
  └─ +烫金+击凸: 印刷 → 烫金 → 击凸 → UV → 裱纸 → 模切 → 糊盒 → 包装
```

**模板选好 → 规则引擎自动校验兼容性 → 微调 → BOM 完成。**

#### 3.3 前加工/后加工自动关联

现有数据已经区分了"前加工工序"和"后加工工序"。建立自动关联规则：

```
前加工选项 → 限制后加工选项
  "过普通膜"   → 后加工可选: 烫金/丝印/UV (但温度受限)
  "过UV膜"     → 后加工可选: UV (不可烫金)
  "不加工"     → 后加工可选: 全部
```

### 第三阶段：AI BOM 编排 + PO 解析（6-8 周）— 25x 提速

**目标：** 输入 PO → AI 推荐完整 BOM，人工确认即可。一个 BOM 从 4h → 10min。

#### 3.1 AI BOM 编排

基于现有 AI Copilot（DashScope）扩展：

```mermaid
输入: "5000 张精装贺卡，157g 铜版纸，烫金+击凸，过哑膜"
  │
  ▼
① IntentDetector 识别：
   类型=贺卡, 工艺=烫金+击凸+过膜, 数量=5000
  │
  ▼
② SkillRouter 匹配工艺模板：
   贺卡_烫金击凸过膜模板
  │
  ▼
③ 从数据库加载：
   - 物料匹配结果（金箔纸 A 适配 ✓）
   - 工序兼容性校验（烫金→击凸→过膜 顺序 OK ✓）
   - 设备分配（烫金机 T-03 可用 ✓）
  │
  ▼
④ 输出 BOM 草案 → 人工确认 → 生产作业单
```

**已具备的 AI 基础设施：**
- AiChatOrchestrator（3 层渐进上下文）✅
- IntentDetector + SkillRouter ✅
- DashScope API 连接 ✅
- 已有物料匹配数据库 ✅

**需要新增的：**
- BOM 模板库（32 类 × 5-8 模板/类 = 约 200 模板）
- 工序兼容性规则表
- BOM 草案推荐 Prompt

#### 3.2 PO 结构化入口

最简单的方式：**做一个订单录入界面，不搞 OCR/NLP**

```
客户 PO (PDF/图片/邮件)
  │
  ├─ 高频客户 → 建产品模板（客户 A 的"贺卡 NO.123" = 已知 BOM）
  ├─ 中频客户 → 下拉选择（品类+工艺+纸张+数量）
  └─ 低频客户 → 自由输入 + AI 解析
```

不需要完美解析 PDF——让跟单员花 2 分钟选参数，比花 2 小时读 PO 做 BOM 快 60 倍。

## 四、技术实现路径

### 4.1 已有资产的复用

| 现有组件 | 在 BOM 系统中的角色 |
|---------|------------------|
| 物料匹配系统（YTS） | 物料兼容性判定引擎 |
| 规则引擎（configurable-rule-engine） | 工序兼容性 + 工艺规则 |
| ETL 管线 | 新品类的数据导入 |
| AI Copilot | BOM 编排推荐 |
| 数据库 36 表 | 物料/兼容性/产品数据 |

### 4.2 需要新增的

| 新增 | 说明 | 工作量 |
|------|------|--------|
| `工序表` (operations) | 32 大类 × 各工序定义 | 1 周 |
| `工序兼容性表` (operation_compatibility) | 工序 vs 工序兼容规则 | 2 周 |
| `BOM 模板表` (bom_templates) | 32 类 × 5-8 典型模板 | 2 周 |
| `BOM 头表` (bom_headers) | BOM 主记录 | 0.5 周 |
| `BOM 明细表` (bom_items) | BOM 行项目 | 0.5 周 |
| `订单录入界面` | PO 结构化入口 | 2 周 |
| AI BOM Prompt | 编排推荐 prompt | 1 周 |

### 4.3 排期

```
Week 1-2   ETL 导入 32 类数据（书籍/包装/贺卡等缺的数据）
Week 3-4   建工序兼容性规则表 + 匹配配置
Week 5-6   建工艺模板库 + BOM 数据表
Week 7-8   订单录入界面 + AI BOM 编排对接
Week 9-10  测试 + 上线 + 回访

总计：10 周
```

### 4.4 预期效果

| 阶段 | BOM 速度 | 提升倍数 | 前提 |
|------|---------|---------|------|
| 当前 | 2 个/天 | 1x | — |
| Phase 1 物料全覆盖 | 6 个/天 | **3x** | 32 类数据入库 |
| Phase 2 工序模板 | 16 个/天 | **8x** | 工艺模板库 |
| Phase 3 AI 编排 | 50+ 个/天 | **25x** | AI BOM + PO 录入 |

## 五、风险与依赖

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| 资深师傅的经验难提炼为规则 | 🔴 | 跟师傅做 5 个典型 BOM 跟单，录成模板 |
| 32 类数据分散，采集耗时 | 🟡 | 优先做高频品类（书籍+包装占 70% 产量） |
| AI 推荐不准确导致返工 | 🟡 | 先做"推荐 + 人工确认"模式，不直接出单 |
| 工序兼容性有隐式约束 | 🟡 | 建规则时留"人工复核"标注位 |
| 排期 10 周资源不足 | 🟡 | 可以先做 3 周 MVP（2 个品类全流程验证） |

## 六、MVP 建议（3 周）

如果资源有限，先做最小验证：

```
选 2 个高频品类：精装书 + 礼盒（占产量 40%+）
  → 3 周做完 → 验证 BOM 速度提升
  → 效果好 → 扩展到 32 类
```

**3 周 MVP 内容：**
1. 精装书 + 礼盒的工序兼容性规则（匹配现有规则引擎）
2. 各 5 个典型工艺模板
3. BOM 录入界面（简单版）
4. 目标：从 2 个/天 → 10 个/天（限精装书+礼盒）
