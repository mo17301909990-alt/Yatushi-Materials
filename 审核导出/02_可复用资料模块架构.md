# 可复用资料模块架构设计

**设计目标**: 构建一个通用的"印刷厂资料管理模块"，能够处理任意类型的工厂文档（烫金纸、过胶材料、UV油、水油、硅胶等），将50+份工厂Excel源文档的处理经验沉淀为可复用架构。

**经验来源**: 已处理16个P0资料模块（106种物料、6470条兼容记录），涉及6大类硅胶产品、UV油哑油、水油哑油、过胶材料、压光UV油。

---

## 1. 核心数据模型

### 1.1 文档模板注册

每种工厂文档类型在入库前必须先注册模板。模板定义了"如何解析-如何存储-如何展示"三要素。

```sql
-- 文档模板注册表
CREATE TABLE IF NOT EXISTS doc_template_registry (
    id                SERIAL PRIMARY KEY,
    template_key      VARCHAR(100) NOT NULL UNIQUE,  -- e.g. 'silicone_glow_ink'
    display_name      VARCHAR(200) NOT NULL,          -- e.g. '硅胶发光油墨'
    category          VARCHAR(50) NOT NULL,            -- 大类: silicone / uv_oil / water_varnish / lamination
    parser_class      VARCHAR(200),                   -- Python parser 类路径
    version           INT DEFAULT 1,                  -- 模板版本号（变更时+1）
    is_active         BOOLEAN DEFAULT TRUE,
    created_at        TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at        TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**Type分类体系**（从50+文档中提炼）:

| Type | 特征 | 示例 | 数量 |
|------|------|------|------|
| A - 标准物料 | 有完整物料编码、规格、用途、颜色 | UV油哑油、水油哑油 | 2个模块，800+产品 |
| B - 硅胶系列 | 同物料名+多种variants，高NULL率 | 发光油墨、白UV、闪星 | 12个模块，50+产品 |
| C - 过胶材料 | 字段重用/污染，responsible_person存厚度 | 过胶材料 | 1个模块，19个产品 |
| D - 极简模板 | 1-2条记录，几乎为占位 | 珠光云母、压光UV油 | 2个模块 |

### 1.2 通用物料模型

从16个模块中提取"最大公约数"字段，同时保留Type-specific扩展通道。

```sql
-- 通用物料主表
CREATE TABLE IF NOT EXISTS generic_material (
    id                SERIAL PRIMARY KEY,
    template_id       INT NOT NULL REFERENCES doc_template_registry(id),
    
    -- 核心标识（所有类型必须）
    material_name     VARCHAR(200) NOT NULL,
    material_code     VARCHAR(100),          -- 可为 NULL（C类经常缺失）
    
    -- 常用分类维度（大部分类型有）
    category          VARCHAR(100),          -- 油類/水類/UV类
    color             VARCHAR(100),
    usage             TEXT,                  -- 用途描述（A/B类最完整）
    
    -- 供应商/库存（A/D类有，B/C类常NULL）
    supplier_code     VARCHAR(100),
    stock_code        VARCHAR(100),
    
    -- 尺寸规格（A/B类核心字段，C类罕见）
    min_sheet_size    VARCHAR(100),
    max_sheet_size    VARCHAR(100),
    min_spacing       VARCHAR(100),
    max_spacing       VARCHAR(100),
    
    -- 质量跟踪
    responsible_person VARCHAR(100),         -- 注意C类曾污染此列
    design_info       TEXT,
    applicable_interface TEXT,
    notes             TEXT,
    
    is_active         BOOLEAN DEFAULT TRUE,
    created_at        TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at        TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 模板特有扩展字段
CREATE TABLE IF NOT EXISTS generic_material_extension (
    id                SERIAL PRIMARY KEY,
    material_id       INT NOT NULL REFERENCES generic_material(id) ON DELETE CASCADE,
    field_key         VARCHAR(100) NOT NULL,   -- e.g. 'film_thickness', 'uv_viscosity'
    field_value       TEXT,
    UNIQUE (material_id, field_key)
);

-- 通用兼容性表（核心复用表）
CREATE TABLE IF NOT EXISTS generic_compatibility (
    id                    SERIAL PRIMARY KEY,
    template_id           INT NOT NULL REFERENCES doc_template_registry(id),
    product_id            INT NOT NULL REFERENCES generic_material(id),
    post_processing_step  VARCHAR(200) NOT NULL,
    compatibility_status  VARCHAR(50) DEFAULT 'V',  -- V / X / ▷
    display_order         INT DEFAULT 0,
    source_file           VARCHAR(500),             -- 追溯来源
    UNIQUE (template_id, product_id, post_processing_step)
);
```

### 1.3 为什么不用纯EAV

EAV（Entity-Attribute-Value）虽然理论上更灵活，但现实中有以下反例:

- **过胶材料**的responsible_person被误存厚度值——如果走纯EAV，这种语义污染更难检测
- **查询性能**: 兼容性矩阵查询涉及大量JOIN和行列转换，EAV在PostgreSQL上的性能无法满足3000+步/模块的实时查询
- **审计追溯**: 需要知道"哪一列存储了什么"，EAV丢失了列级语义约束

所以采用 **"通用强类型主表 + 扩展KV表"** 的混合方案。主表保留所有已知字段的强类型定义（foreign key、CHECK约束、NOT NULL where applicable），扩展表处理每个Type特有的字段。

### 2.2 Parser Plugin 接口

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import List, Dict

@dataclass
class ParsedProduct:
    material_code: str = None
    material_name: str = None
    category: str = None
    color: str = None
    usage: str = None
    # ... 其他通用字段
    extensions: Dict[str, str] = None  # Type-specific

@dataclass  
class ParsedCompatibility:
    product_id: int  # 解析阶段用行号占位
    post_processing_step: str
    compatibility_status: str  # V/X/▷

@dataclass
class ParseResult:
    products: List[ParsedProduct]
    compatibilities: List[ParsedCompatibility]
    warnings: List[str]
    errors: List[str]

class BaseParser(ABC):
    """所有Parser必须实现此接口"""
    
    @abstractmethod
    def detect(self, filepath: str) -> float:
        """返回0-1匹配度，用于自动模板匹配"""
        pass
    
    @abstractmethod
    def parse(self, filepath: str) -> ParseResult:
        """解析文件返回结构化数据"""
        pass
    
    def validate(self, result: ParseResult) -> List[str]:
        """校验规则（可覆写）"""
        warnings = []
        # 默认规则
        for p in result.products:
            if not p.material_name:
                warnings.append(f"行{p}: material_name为空")
            if p.usage and len(p.usage) > 500:
                warnings.append(f"行{p}: usage过长({len(p.usage)}字符)")
        return warnings
```

### 2.3 通用校验规则引擎

从50+文档的实战中总结的校验规则:

```python
VALIDATION_RULES = {
    "compatibility_status": lambda v: v in ("V", "X", "▷"),
    "material_code_format": [
        lambda v: bool(re.match(r'^[#A-Za-z0-9\-_/]+$', v)) if v else True,
        lambda v: len(v) <= 100 if v else True,
    ],
    "sheet_size": lambda v: not v or bool(re.match(r'^\d+(\.\d+)?\s*[×xX*]\s*\d+(\.\d+)?', v)),
    "layer_2_field_consistency": {
        # 中文异体字检测（从繁简混用Bug总结）
        "post_processing_step": lambda v: not any(c in v for c in "燙曬"),
    },
}
```

**重点规则**（从0/6 API空结果中学习）:

1. **Null值级联判断**: 如果 category/usage/color 全部为NULL，则标记整个物料为"不可筛选"，生成告警
2. **字段污染检测**: 如果 responsible_person 列的值匹配 `^\d+(\.\d+)?(mm|絲|um)?$`，则触发"疑似字段误用"告警
3. **跨表参照完整性**: 兼容性表中的product_id必须在产品表中存在（已有约束，但需要在预览阶段预检）

### 2.4 Parser应对已知陷阱

| 陷阱 | 表现 | 应对策略 |
|------|------|---------|
| 繁简混用 | "烫/燙"、"晒/曬"并存 | Parser自动统一成简体（转换表），同时写入变更日志 |
| 合并单元格 | openpyxl读取合并单元格只有左上角有值 | 使用openpyxl的`merged_cells`属性填充所有被合并的行 |
| 换行符/尾空格 | Excel单元格内换行导致SQL存储差异 | Parser阶段strip() + replace('\n','') |
| 单位后缀不一致 | "0.03mm" vs "0.03" vs "3絲" | Parser统一标准化输出 |
| 空行/分隔行 | Excel中的空行或"---"分隔行 | 跳过所有空行 + 只含特殊字符的行 |
| 重复表头 | 多sheet文档中每个sheet有独立表头 | 自动检测并跳过与第一行表头匹配的行 |

## 4. 质量保证

### 4.1 上传自动检测

```python
def auto_detect_duplicates(parsed: ParseResult, template_id: int) -> List[DuplicationWarning]:
    """上传时自动检测三类重复"""
    warnings = []
    
    # 1. 文件级重复 (same SHA256)
    existing = db.query("SELECT id, file_hash FROM data_import_audit_log WHERE template_id = %s", template_id)
    for e in existing:
        if e.file_hash == sha256(filepath):
            warnings.append(f"该文件已于 {e.created_at} 导入过 (id={e.id})，确认是否重新导入？")
    
    # 2. 物料级重复 (same material_code + template_id)
    for p in parsed.products:
        if p.material_code:
            existing = db.query(
                "SELECT id, material_name FROM generic_material WHERE template_id = %s AND material_code = %s",
                template_id, p.material_code
            )
            if existing:
                warnings.append(f"物料编码 {p.material_code} 已存在 (现有: {existing[0].material_name})，将执行覆盖")
    
    # 3. 跨模板重复（跨模块共享物料）
    # e.g. #CS-220902 同时出现在 glittering_star 和 screen_uv
    for p in parsed.products:
        if p.material_code and p.material_code.startswith('#'):
            cross_refs = db.query("""
                SELECT t.display_name, m.material_name 
                FROM generic_material m 
                JOIN doc_template_registry t ON m.template_id = t.id
                WHERE m.material_code = %s AND m.template_id != %s
            """, p.material_code, template_id)
            if cross_refs:
                warnings.append(f"跨模块提醒: 物料 {p.material_code}({p.material_name}) 也存在于 {cross_refs[0].display_name}")
    
    return warnings
```

### 4.2 交叉引用校验

每次新数据导入后，自动执行交叉引用完整性检查:

| 检查项 | SQL | 严重度 |
|-------|-----|--------|
| 孤立兼容记录 | `SELECT * FROM generic_compatibility c LEFT JOIN generic_material m ON c.product_id=m.id WHERE m.id IS NULL` | ERROR（阻止提交） |
| 无兼容产品 | `SELECT m.* FROM generic_material m LEFT JOIN generic_compatibility c ON m.id=c.product_id WHERE c.id IS NULL` | WARNING |
| V/X比例异常 | `SELECT template_id, compatibility_status, COUNT(*) FROM generic_compatibility GROUP BY 1,2` | INFO（统计参考） |

### 4.3 审批工作流

```
[上传] → [自动校验] → [预览] → [提交审批] → [审批通过] → [数据生效]
                          │                    │
                          ↓                    ↓
                     [拒绝/退回]           [退回修改]
```

- **低风险操作**（新增记录 < 10条）：自动审批
- **中风险操作**（修改/删除）：需1级审批
- **高风险操作**（全量替换/删除大量记录）：需2级审批 + 快照备份

## 5. 当前技术栈映射

### 5.1 ETL (Python)

| 组件 | 选用 | 原因 |
|------|------|------|
| Excel解析 | openpyxl (v3.1.2) | 同时支持读取和写入，中文编码处理稳定 |
| DB连接 | psycopg2 / psql CLI | 当前导出脚本使用psql CLI（`-A -F\t`），长期建议迁移到psycopg2连接池 |
| 数据清洗 | pandas（可选） | 对大型兼容性矩阵的行列转换有用，但小模块推荐纯Python dict处理以提高可追踪性 |
| 配置管理 | YAML或TOML | 每个模板的字段映射、校验规则、默认值写在单独配置文件中 |

**经验教训**:

- `openpyxl`读取合并单元格时只有左上角有值，需要调用`ws.merged_cells.ranges`填充其他单元格
- 中文编码在Windows psql CLI下容易触发GBK解码错误，必须设置`PGCLIENTENCODING=UTF8`
- 物料表中的"spacing"字段在不同Excel中可能叫法不同（间距/牙距/間距），需要字段映射表

### 5.2 数据库 (PostgreSQL)

| 层次 | 方案 | 说明 |
|------|------|------|

| 序列 | SERIAL | 当前使用，稳定可靠 |

## 6. 实战经验总结

### 6.1 已踩过的坑（务必记录）

1. **繁简混用**: 烫金类型4对值被当作不同值存储，导致1365条数据查询断裂。修复耗费约4倍于预防成本。
2. **字段污染**: 过胶材料的responsible_person列被存为厚度值，造成整个A/B测试阶段数据不可用。
3. **计数不一致**: Parser按"distinct name"计，入库按"行"计（不同usage视为不同行），导致7个模块计数偏差。
4. **API空结果**: 入库数据正确但API返回空——因为WHERE条件中的筛选字段在数据中全为NULL。这是"数据对但业务不对"的典型案例。
5. **模板空占位**: mica_pearl和yaguang_uv_oil两个模块各自只有1条记录，没有实际业务价值但仍被计入P0。

### 6.2 推荐演进路线

```
Phase 1 (当前): 16个独立模块架构 → 验证业务正确性
Phase 2 (2周): 实现 generic_material + generic_compatibility 通用表
Phase 3 (1月): 完成统一上传/预览/导入Pipeline
Phase 4 (2月): 前端模板管理和上传向导
Phase 5 (3月): 审批工作流 + 自动交叉校验
```

### 6.3 设计原则

1. **优先"对齐"而非"灵活"**: 强类型主表优于纯EAV。扩展字段用KV表弥补，但不做为主数据通道。
2. **上传即校验，不做静默转化**: 所有数据冲突在上传预览阶段展示给用户，不静默覆盖或丢弃。
3. **每列可追溯**: 每个字段都知道"来自哪个Excel的哪一列"，便于回查。
4. **审计第一**: 导入日志记录每一次变更的前后状态，"谁在什么时间改了哪个值"。
5. **兼容性是核心资产**: 兼容性数据（V/X/▷）是印刷厂数字化最有价值的资产，需要比物料数据更高的保护级别。
